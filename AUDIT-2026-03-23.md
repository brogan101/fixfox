# FixFox Audit - 2026-03-23

## Scope

This audit focused on:

- icon rendering paths
- embedded logo resources
- WPF resource integrity
- fix catalog health
- obvious project-level architecture gaps that affect usability and future feature work

## What Was Checked

- 16 `ui:SymbolIcon` usages in XAML
- 8 `IconFont`/MDL2 glyph usages in XAML
- embedded `FixFoxLogo.png` and `FixFoxLogo.ico` resources
- startup verification output from the built app artifact
- fix catalog counts, bundle references, and converter instantiation

Verification command used after the fixes in this audit:

```powershell
dotnet .\bin\Debug\net8.0-windows\FixFox.dll --verify-headless
```

Result after fixes:

- 24 passed

## Icon Audit Result

### Pass

- No broken `SymbolIcon` names were found in compiled XAML.
- The app logo resources are embedded correctly.
- Theme resources are compiled correctly.
- The built app now verifies cleanly in headless mode.

### Fixed During Audit

1. Dashboard scan results were binding `ScanSeverity` into `StatusIconConverter`, which is a fix-status converter, not a scan-severity converter.
   Outcome: added a dedicated `SeverityIconConverter` and switched the dashboard to use it.

2. Headless verification was producing false failures for the logo and theme checks because it relied on UI-only resource loading paths.
   Outcome: changed verification to inspect compiled WPF resources directly from the assembly manifest.

3. Headless verification could throw when trying to force UTF-8 output in shells without a valid console handle.
   Outcome: wrapped the encoding change so verification never fails for that reason.

## Key Findings

### High confidence

1. Icon pipeline is healthy after the fixes above.
   Evidence: build succeeds, all compiled resource checks pass, and the built artifact reports `24/24` successful startup checks.

2. The project has strong catalog depth already.
   Evidence: 213 fixes across 21 categories and 16 bundles.

3. There is service-registration drift between the UI startup path and the headless startup path.
   `App.xaml.cs` registers `DuplicateFileService`, `InstalledProgramsService`, and `SchedulerService`, but `Program.BuildServices()` does not.
   This is not an icon bug, but it is a real architecture inconsistency that can block future automation and test coverage.

### Medium confidence / watch list

1. Several source files display mojibake in the terminal for some comments and strings.
   The built artifact rendered and verified correctly, so this currently looks like a source/terminal encoding presentation issue more than a runtime icon issue.

2. `dotnet run --project HelpDesk.csproj -- --verify-headless` triggered a WPF temp-project build issue in this environment.
   The compiled artifact verification path works correctly and should be preferred for audit/CI style checks.
