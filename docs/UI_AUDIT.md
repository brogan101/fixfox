# UI Audit (Pre-Edit)

Date: 2026-03-05

## Navigation
- Main app nav is rail-only in structure (`NavRail`), but it is still placeholder-glyph based:
  - `src/ui/main_window_impl.py::_build_nav` maps nav IDs to text glyphs (`"^"`, `"EX"`, `"*"`, etc.).
  - `src/ui/components/nav.py::_make_button` uses `QToolButton` text-only mode and renders glyph text, not real icons.
- A second nav-like pattern exists inside Settings (expected for settings sub-sections), but visual language collides with rail rows:
  - `src/ui/pages/settings_page.py` uses `QListWidget#SettingsNav`.
  - `src/ui/main_window_impl.py::_settings_nav_item_widget` reuses `_rail_row_widget`, so settings rows look like rail rows and are prone to cramped layout.

## App Shell Layout / Splitter Behavior
- Shell structure is in `src/ui/components/app_shell.py::AppShellFrame` with a 3-pane `QSplitter`:
  - left rail, center page host, right side sheet.
- Right pane behavior is controlled in:
  - `src/ui/main_window_impl.py::_set_concierge_collapsed`
  - `src/ui/main_window_impl.py::_apply_responsive_concierge`
- Side pane is not consistently treated as contextual-only in all states:
  - `set_ui_mode("basic")` had an early-return path that could leave pane state stale (recently mitigated, but architecture is still fragile around persisted layout + mode interactions).
- Splitter handle visual can read as overly accent-heavy:
  - `src/ui/style/qss_builder.py` uses accent-tinted handle hover/pressed (`QSplitter::handle:*`), which can appear as random orange highlights depending on palette.

## Global Styling Coverage
- Canonical stylesheet is generated by `src/ui/style/qss_builder.py` and applied in:
  - `src/app.py` (`app.setStyleSheet(build_qss(...))`)
  - `src/ui/main_window_impl.py::_apply_theme`
- Styling is comprehensive for named controls, but onboarding misses much of this due generic widgets:
  - `src/ui/components/onboarding.py` uses plain `QPushButton/QLabel/QComboBox` without object names used by QSS selectors.
  - Result: onboarding can look low-contrast/unpolished compared with main shell.

## Icons
- Current icon system is fallback-only:
  - `src/ui/icons.py` returns `QStyle.StandardPixmap` icons.
  - `src/assets/icons/` is effectively empty (`.gitkeep` only).
- Placeholder text icons still exist in shell:
  - App bar buttons use text (`"S"`, `"EX"`, `"..."`) in `src/ui/components/toolbar.py`.
  - Side sheet pin/close use text (`"PIN"`, `"X"`) in `src/ui/components/app_shell.py`.

## Search Wiring
- Search data flow is present and connected:
  - `top_search` -> `_schedule_global_search` (debounced) -> `_refresh_global_search_results`
  - `query_index(...)` in `src/core/search.py`
  - `GlobalSearchPopup.show_results(...)` in `src/ui/components/global_search.py`
  - selection dispatch via `_apply_global_search_result` -> `_dispatch_search_selection`
- Practical UX gaps:
  - compact-search button opens command palette directly instead of search popup path.
  - app bar placeholder/hinting does not explicitly communicate `Ctrl+K` path.
  - popup readability depends on generic list styling and needs stronger polish consistency.

## Settings
- IA duplication exists:
  - both `Privacy` and `Privacy & Safety` sections in `src/ui/pages/settings_page.py`.
- Left settings subnav can look crowded:
  - fixed min/max width (`200/280`) + custom item widgets via `_settings_nav_item_widget`.
  - row rendering currently reuses rail-row text pattern and does not include clear section iconography.
- Persistence wiring is real and mostly correct:
  - values bind through `save_settings_from_ui` and `_sync_settings_ui` in `src/ui/main_window_impl.py`.

## Branding
- Top identity block is in `src/ui/components/toolbar.py`:
  - mark from `assets/branding/fixfox_mark.svg`
  - wordmark via `APP_DISPLAY_NAME`
  - status chip updated in `src/ui/main_window_impl.py::_set_run_status`
- Branding issues observed:
  - mark rendering can appear inconsistent depending on SVG/background context.
  - iconography around brand block is text placeholders rather than brand-consistent symbols.

## Font Loading / DejaVu Error Root Cause
- No active `DejaVuSans.ttf` usage found in current runtime path.
- Font load path is centralized in `src/app.py`:
  - `_font_candidates(...)`
  - `_load_bundled_font(...)` using `QFontDatabase.addApplicationFontFromData`.
- Current implementation validates font blob and falls back to Segoe UI cleanly; no CWD-relative DejaVu pathing present.

## User-Facing Qt Branding Leakage
- `About Qt` is still surfaced in overflow menu:
  - `src/ui/main_window_impl.py::_open_header_overflow_menu` adds `"About Qt"` action.
- This violates the current requirement to remove Qt branding from user-facing UI.
